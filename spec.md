# PART I — DESIGN

## 1. Overview & High-Level Idea

Build a multi-agent Agentic AI system that accepts deployment parameters such as radio hardware resources, geographic area, expected user density, traffic profile, spectrum bands, latency constraints, compute resources, and deployment budget, for a specific geographic area — Malleswaram, North Bangalore — and automatically plans, configures, deploys, or modifies an Open RAN (O-RAN) compliant 4G/5G network. The system is context-aware of the live deployment and continuously optimizes for load balance and operator efficiency. The live deployment is simulated by 3 DU containers and 1 CU container covering a population of 40,000 residents + 15% commuter overhead (46,000 effective), with 18,400 peak active UEs at 40% operator market share. All UE telemetry is synthetic, generated by the DU simulators using realistic hardware specifications and diurnal load curves.

- Orchestrator — Understands operator commands with the use of an LLM and routes them to the right tools and gives appropriate response back to the operator.
- Controller — the single source of truth for the network topology; handles live cell/DU moves
- Planning Agent — designs new network layouts for cell placement and DU/CU grouping or if at all new cell placement is required for the increasing demand, UE counts and hardware limits are calibrated to real market equipment specs using placement algorithms, PCI planning, and slice allocation.
- KPI Agent — Continuously monitors cell health using a bidirectional LSTM model, detects anomalies (overload, poor signal, power waste), and takes autonomous corrective actions
- Map Server — A live map container shows the current cells layout with per-cell KPI overlays, colour-coded by vendor and generation.

The RAN is a 4G/5G NSA mix — both 4G LTE and 5G NR connect to the same NSA core (shared AMF/SMF/UPF). 5G NR deployment uses split CU/DU/RU architecture throughout;

The KPI agent, controller, or planning engine can reorganise cells at any time in response to changing load or operator instruction. Input may be partial — the system will request missing fields or flag redundant ones.

## 5. Network Topology — Malleswaram

**Population model:** 40,000 residents + 15% commuter overhead (railway station transit hub) = 46,000 effective. Tier-1 operator 40% market share → **18,400 peak active UEs**. Busy-hour traffic = 18,400 × 0.025 Erlangs/user = **460 Erlangs**. At 16 Erlangs per sector (Erlang-C, 2% blocking) → **29–30 sectors required**.

| CU | DU | Sites served | Cells (initial design) |
|---|---|---|---|
| CU-MLS | DU-MLS-1 | RWS, 18C, BEL, SNK (north) | 12 |
| CU-MLS | DU-MLS-2 | SPG, 3MN, 10C (central) | 9 |
| CU-MLS | DU-MLS-3 | MGR, CHD, 6CR (south-west) | 9 |

> The "Cells" column is the **initial design** grouping. DU↔cell assignment is runtime
> state mutated by `move_cell` / SON, so the live `topology.json` snapshot differs.
> 📖 [Appendix A](#appendix-a--data-model).

**30 cells (10 macro sites × 3 sectors). 700 MHz (n28) not deployed — coverage radius (~8.4 km) extends beyond Malleswaram to Peenya. Sector mix per site:**

| Sites | Sector 1 | Sector 2 | Sector 3 |
|---|---|---|---|
| RWS, 18C, SNK, SPG, 10C (high traffic) | 5G n78 3500 MHz | 5G n41 2500 MHz | 4G B3 1800 MHz |
| BEL, 3MN, MGR, CHD, 6CR (residential) | 5G n78 3500 MHz | 4G B40 2300 MHz | 4G B3 1800 MHz |

**Cell-level max UEs: 16,500 (900×10 n78 + 700×5 n41 + 300×5 B40 + 250×10 B3). active_ues_peak: 18,400 (small headroom gap absorbed by Erlang-C blocking tolerance).**

## 6. Agent Responsibilities

This section describes each agent's behaviour. Exact constants, schemas, and

### 6.1 Orchestrator (`agents/orchestrator/`)

FastAPI service on port 8082. Accepts natural-language operator commands over HTTP,
drives a multi-step tool-calling loop, and streams the response back in real time.
Default LLM backend is **Google Gemini** (`google-genai` SDK); an optional
**Claude-CLI** backend is selectable via env var.

> 📖 Endpoints, request/response: [Appendix F.3](#f3-orchestrator).
> Backend selection (Gemini / Claude CLI) and config: [Appendix G](#appendix-g--orchestrator-llm-backends).

**Request / response flow**

```
User message  (POST /chat)
      │
      ├─► build_network_context()  ──GET /network──► Controller
      │         (live cell snapshot appended to system prompt)
      │
      ▼
SYSTEM_PROMPT + live snapshot + session history
      │
      ▼
  ┌──────────────────────────────────────────────┐
  │   LLM  (non-streaming generate call)         │
  └──────────────────────────────────────────────┘
      │
      ├── text parts → yield to caller (streaming)
      │
      └── function_call parts → tool-calling loop:
              ├─ yield "*[calling tool: name...]*"
              ├─ execute_tool(name, args)  (Python call)
              ├─ append FunctionResponse to history
              └─ call LLM again  →  repeat until no tool calls
```

**System prompt** — two parts injected every request:
- **Static** (`SYSTEM_PROMPT`): 30-cell overview — naming (`MLS_<SITE>_<SECTOR>`), DU/CU hierarchy, per-band UE limits and power specs, operator guidelines (confirm before destructive actions, flag overloads, bullet summaries).
- **Dynamic** (`build_network_context()`): `GET /network` and formats every cell as one line `cell_id (area) → DU=... | UEs=... | PRB=...% | SINR=...dB | Power=...W`. Returns a warning if the Controller is unreachable.

**Tool-calling loop** — each `/chat` runs a `while True` loop until the model returns no function calls: generate → append assistant turn → yield text parts → for each `function_call`: yield a `*[calling tool]*` marker, call `T.TOOL_MAP[name](args)` (synchronous HTTP to Controller / Planning / InfluxDB), JSON-sanitise the result, append a `FunctionResponse` user turn → repeat. Multiple tools may be batched per response and are all executed before the next model call.

**Tool inventory**

| Tool | HTTP call | Purpose |
|---|---|---|
| `query_network` | `GET /network` on Controller | Full topology + live KPIs for all 30 cells |
| `list_cells` | `GET /cells?area=&du_id=&cu_id=` | Filtered cell list with KPIs |
| `query_cell` | `GET /cells/{id}` | Single cell config + 30-min KPI time series |
| `move_cell` | `POST /move/cell` | Reassign a cell to a different DU |
| `move_du` | `POST /move/du` | Reassign a DU to a different CU |
| `plan_network` | `POST /plan` | Heuristic or MIP-optimal placement + PCI + slice planning |
| `plan_network_multi_period` | `POST /plan/multi-period` | Multi-period MIP (Case A phased rollout / Case B diurnal shift) |
| `apply_plan` | `POST /plan/apply` | Push accepted plan to Controller as live topology |
| `get_alerts` | InfluxDB Flux query (direct) | Recent KPI anomaly alerts tagged by severity and type |
| `query_ue` | InfluxDB Flux query (direct) | UE-level usage and mobility data (filter by ue_id or cell_id) |
| `get_son_status` | InfluxDB Flux query (direct) | SON action summary + counts by type, last 10 actions, active alert severity |
| `add_cell` | `POST /cells/add` on Controller | Deploy a new cell via chat; auto-assigns PCI if not provided |
| `remove_cell` | `DELETE /cells/{id}` on Controller | Decommission a cell and remove from DU assignment |

**Tool schema translation** — `tools.py` stores all 13 schemas in Anthropic-style JSON (`name`, `description`, `input_schema`). At startup `_clean_params()` translates them to Gemini `function_declarations`: strips `default` fields, removes empty `enum` arrays (the `""` sentinel in `severity`), and deep-copies to avoid mutating `TOOL_SCHEMAS`. `GEMINI_TOOLS` = `[{"function_declarations": [...]}]`.

**Sessions** — in-memory per `session_id`; Gemini stores `types.Content`, Claude stores `{role,content}` dicts. `DELETE /history` clears a session; lost on restart. `GET /history` normalises to `{"role","content"}` (tool calls shown as `[Calling name]`, results `[Tool result: name]`).

**Streaming** — `POST /chat` returns a `StreamingResponse` wrapping a synchronous generator (`chat_turn`); Starlette runs it in a thread pool so blocking LLM/tool calls don't stall the event loop. Chunks: one per LLM turn; `*[calling tool: name...]*` before each tool; `\n\n[Error] ...` on quota/rate-limit (`"429"`, `"quota"`, `"ResourceExhausted"`).

### 6.3 Planning Engine (`agents/planning/`)

- FastAPI on port 8081
- `POST /plan` — deployment parameters → complete network plan; `use_mip=true` for MIP-optimal placement
- `POST /plan/multi-period` — multi-period MIP planning (Case A phased rollout / Case B diurnal/event shift)
- `GET /demand-clusters` — pre-defined Bangalore demand clusters + preset period profiles
- `POST /plan/apply` — pushes accepted plan to Controller as topology update
- Algorithms: density-weighted heuristic (default) OR MIP-optimal placement (Almoghathawi et al. 2024), graph-coloring PCI planner, PRB slice allocator, distance-based fronthaul/midhaul router

> 📖 Algorithm constants (PCI, slice, placement, MIP): [Appendix E](#appendix-e--planning-engine-constants).
> Endpoints, request/response: [Appendix F.2](#f2-planning-api).

**Propagation models**

| Model | Used by | Environment | Notes |
|---|---|---|---|
| COST-231-Hata | DU/CU simulators, Map Server | Urban macro, empirical | Coverage radius, KPI simulation |
| **COST-231-Walfisch-Ikegami** | **MIP placer** | **Urban NLOS** | **Path loss for link-budget feasibility and SINR constraints** |

**MIP-based placement (Almoghathawi et al., 2024)** — Reference: Almoghathawi Y., Bin Obaid H., Selim S. — *"Optimal location of base stations for cellular mobile network considering changes in users locations"*, Journal of Engineering Research 13 (2025) 561–567. DOI: 10.1016/j.jer.2024.04.020

Formulation — MIP minimising total network cost:

```
Minimise  Σ_j Σ_t ( c_jt·z_jt  +  r_jt·y_jt )
```

where `z_jt` = build BS at site j in period t (one-time CAPEX), `y_jt` = BS active in period t (per-period OPEX). Subject to:
- **(2) Single-build**: each candidate site built at most once across all periods
- **(3) Coverage**: every demand cluster served by a BS built in this or an earlier period
- **(4) Activation**: BS can operate only after it has been built
- **(5) Unique assignment**: each demand cluster assigned to exactly one BS per period
- **(6) Implies-active**: assigning demand to a site forces that site active
- **(7) Capacity**: channel demand at each BS ≤ δ_j (max UE capacity)
- **(8) SINR QoS**: received SINR at each demand cluster ≥ SINR_min; linearised as `α(i,t)·(1 + SINR_lin) ≥ SINR_lin·P_noise + SINR_lin·β(i,t)`

**Demand node concept** (Tutschku 1998): traffic per area is a finite set of *demand clusters* D, each with a channel requirement ρ_i, separate from candidate sites S; **3 pre-defined Malleswaram demand clusters** (DC-MLS-N / DC-MLS-C / DC-MLS-S), one per DU service zone (see Appendix E.4).

**Multi-period modes**: **Case A — permanent/expanding** (each period adds demand; BSs built early serve later periods, avoiding redundant spend); **Case B — temporary/shifting** (demand shifts between periods: residential → IT-hub → evening commute; solver minimises total cost across scenarios).

**Cost model**: `install_cost` (c_jt) one-time CAPEX; `op_cost` (r_jt) per-period OPEX; both vary by location/period. **Solver**: CBC via `pulp`; falls back to heuristic on timeout/infeasibility.

### 6.4 KPI Monitoring Agent (`agents/kpi_agent/`)

- Background process (no exposed port)
- Polls InfluxDB; maintains a 6-step (60 s) sliding window per cell
- Bidirectional LSTM classifier (5 classes): NORMAL, OVERLOAD, UNDERLOAD, SINR_LOW, POWER_WASTE
- 9 features: prb_dl_pct, sinr_db, connected_ues, power_w, packet_loss_pct, dl_throughput_mbps, cqi, bler_pct, latency_ms
- Rule-based fallback while the window fills; then AI inference with a confidence gate
- **Autonomous SON actions** (all 4 anomaly types): OVERLOAD → cell-move to lightest DU + `LOAD_BALANCE`; UNDERLOAD → `TRAFFIC_STEER`; SINR_LOW → `PCI_REOPT_REQUEST` (+ best-effort Controller call); POWER_WASTE → `DTX_RECOMMEND`
- Writes `alerts` and `son_actions` measurements to InfluxDB

> 📖 Model, training, thresholds, SON logic & vocabularies: [Appendix D](#appendix-d--kpi-agent-model-training--son).
> Input features are produced by the DU simulator: [Appendix C.1](#c1-du-simulator).

## Appendix A — Data Model

### A.1 `topology.json` top-level structure

```jsonc
{
  "version":      472,                         // int, incremented on every Controller write
  "last_updated": "2026-06-11T15:59:49.108989+00:00",  // ISO-8601 UTC
  "updated_by":   "move_cell:MLS_SPG_01",      // who/what made the last write
  "meta":  { ... },                            // see A.2
  "cus":   { "<CU_ID>":   { "host", "region", "du_ids":  [ ... ] } },
  "dus":   { "<DU_ID>":   { "cu_id", "host",  "cell_ids":[ ... ] } },
  "cells": { "<CELL_ID>": { ...per-cell fields, see A.3 } }
}
```

- The Controller is the **only** writer (atomic `.tmp` → rename). All simulators are read-only.
- `dus[*].cell_ids` is **runtime state** — mutated by `move_cell` / SON load-balancing.
  A **fresh build ships the §5 initial design (12/9/9)** — the canonical seed file is fully
  specified in [A.5](#a5-canonical-initial-topologyjson-seed). Once the system runs, SON moves
  mutate `cell_ids`, so a *running* system's snapshot (e.g. the 9/6/15 grouping seen in
  long-lived dev volumes) legitimately differs from the seed. `version` starts at 1 in the seed
  and is incremented by 1 on every Controller write.
  Per-cell attributes (`vendor`, `band`, `pci`, coords, power) are intrinsic and stable
  across moves; only DU assignment changes.

## Appendix C — Simulation Models

All simulators push every `INTERVAL_SEC = 10 s`. Common env vars (see [Appendix H](#appendix-h--deployment-compose-images-provisioning)
for the full compose wiring): `INFLUX_URL`, `INFLUX_TOKEN`, `INFLUX_ORG`, `INFLUX_BUCKET`,
`INTERVAL_SEC`. DU/CU simulators additionally read `TOPO_POLL_SEC` (5 s) and `TOPOLOGY_FILE`
(`/config/topology.json`, mounted read-only).

**Simulator identity & topology binding.** Each DU simulator is given a `DU_ID` (`DU-MLS-1/2/3`)
and the CU a `CU_ID` (`CU-MLS`). On every poll the simulator re-reads `topology.json` and serves
exactly the cells currently listed under its id — so SON moves and `/plan/apply` swaps take effect
live. **Naming after `/plan/apply`:** the simulators **follow whatever DU/CU ids the applied plan
writes** (a regenerated plan may introduce `DU-BLR-NN` / `CU-BLR-NN`, see [E.3](#e3-heuristic-placement--grouping-placementpy));
a deployment running the planner is expected to run one DU container per plan-emitted `DU_ID`.
The Core simulator has no topology dependency (see [C.3](#c3-core-simulator)).

**Band note (`n28`):** `n28` (700 MHz) is fully parameterised below but **not deployed** in the
Malleswaram seed (its ~8.4 km radius overshoots the zone). It exists so `add_cell`/plan requests
may select it; no seed cell uses it.

### C.1 DU simulator (`dev-env/simulators/du/du_simulator.py`)

**Coverage radius — COST-231-Hata (Urban Macro).** Fixed: BS height `hb = 25 m`,
UE height `1.5 m`, dense-urban correction `+3 dB`, UE noise figure `7 dB`,
coverage-edge SNR threshold `−3 dB`.

Per-band RF parameters (`_BAND_PARAMS`):

| Band | freq_mhz | bw_mhz | pen_loss_db |
|---|---|---|---|
| n78 | 3500 | 100 | 20 |
| n41 | 2500 | 80 | 20 |
| n28 | 700 | 20 | 15 |
| B3 | 1800 | 20 | 18 |
| B40 | 2300 | 20 | 18 |

Antenna gain `_ANT_GAIN`: 64T64R = 24.0 dBi, 4T4R = 17.0 dBi.
RF efficiency `_RF_EFF`: 5G = 0.22, 4G = 0.32 (default 0.25).

```
rf_w      = max(tx_power_w · rf_eff, 0.1)
eirp_dbm  = 10·log10(rf_w · 1000) + ant_gain
noise_dbm = −174 + 10·log10(bw_mhz · 1e6) + 7.0
pl_max    = eirp_dbm − (noise_dbm − 3.0) − pen_loss_db
A         = 46.3 + 33.9·log10(freq_mhz) − 13.82·log10(25) + 3.0
B         = 44.9 − 6.55·log10(25)            # ≈ 35.74
radius_m  = 10^((pl_max − A) / B) · 1000
```

**Population / demand model.** `AREA_DENSITY["Malleswaram"] = 9000` people/km²
(default 700 elsewhere); `MARKET_SHARE = 0.40`; `PEAK_CONCURRENT = 0.40`.

```
expected_peak_ues = min( π·(radius_m/1000)² · density · 0.40 · 0.40 , max_ues )
```

**Diurnal load curve** `HOURLY_LOAD` (fraction of peak, index = hour 0–23):

```
0.08 0.06 0.05 0.05 0.06 0.12   0.32 0.68 0.88 0.82 0.72 0.66
0.64 0.60 0.62 0.68 0.78 0.90   0.95 1.00 0.97 0.88 0.62 0.28
```

`WEEKEND_FACTOR = 0.75` applied on Saturday/Sunday.
`load_factor() = HOURLY_LOAD[hour] (× 0.75 on weekends)`.
Slice mix per UE (`SLICES`): 7× eMBB, 2× URLLC, 1× mMTC (random per UE).

**Per-tick UE count.** `target_ues = clamp(int(expected_peak_ues · load_factor · U(0.88,1.05)), 0, max_ues)` (initial pool uses `U(0.8,1.0)`). `load = connected_ues / max_ues`.

**`cell_kpi` field formulas:**

```
prb_dl_pct       = min(98, load·100·U(0.92,1.08))
prb_ul_pct       = min(95, load·58 ·U(0.88,1.12))
dl_throughput    = prb_dl_pct/100 · peak_dl · U(0.82,1.18)
ul_throughput    = prb_ul_pct/100 · peak_dl · 0.22 · U(0.80,1.20)
sinr_db          = sinr_base − load·15 + N(0,2.5)
rsrp_dbm         = rsrp_base − load·22 + N(0,3.0)
rsrq_db          = clip(−10 + sinr_db·0.3 + N(0,1.5), −19.5, −3.0)
cqi              = clip(int((sinr_db+5)/2.5 + N(0,0.8)), 0, 15)
mcs              = clip(int(cqi·1.8 + N(0,1.2)), 0, 28)
bler_pct         = max(0, (load−0.75)·15 + (10−cqi)·0.5 + N(0,0.5))
power_w          = max(idle·0.90, idle + load·(tx−idle) + N(0, tx·0.025))
packet_loss_pct  = max(0, (load−0.75)·2.5 + N(0,0.05))
latency_ms       = max(1, 8 + load·25 + max(0,5−sinr_db)·2 + N(0,2))
jitter_ms        = max(0.1, latency_ms·U(0.05,0.15) + N(0,0.3))
interference_dbm = −100 + load·20 + N(0,3)
ho_success_rate  = U(0.962, 0.9995)
```

Per-band baselines:

| Band | `_SINR_BASE` (dB) | `_RSRP_BASE` (dBm) |
|---|---|---|
| n78 | 22.0 | −72 |
| n41 | 20.0 | −74 |
| n28 | 29.0 | −64 |
| B3 | 26.0 | −69 |
| B40 | 23.0 | −73 |

**`du_kpi` formulas** (`load` = Σ connected_ues / Σ max_ues over the DU's cells):

```
active_ues           = Σ connected_ues
cell_count           = number of cells on DU
cpu_pct              = 20 + load·62 + N(0,3)
```

## Appendix D — KPI Agent: Model, Training & SON

Sources: `agents/kpi_agent/model.py`, `train.py`, `kpi_agent.py`.

### D.1 `KPIClassifier` architecture

```
LSTM(input=9, hidden=64, num_layers=2, batch_first=True, dropout=0.25, bidirectional=True)
head = Sequential(
    Linear(128, 64),   # 128 = hidden·2 (bidirectional)
    ReLU(),
    Dropout(0.25),
    Linear(64, 5),
)
forward(x): out,_ = lstm(x); return head(out[:, -1, :])   # last timestep only
```

`SEQ_LEN = 6` (6 × 10 s = 60 s window), `N_FEATURES = 9`, `N_CLASSES = 5`.
Classes: 0 NORMAL, 1 OVERLOAD, 2 UNDERLOAD, 3 SINR_LOW, 4 POWER_WASTE.

### D.2 Feature order & normalisation (`FEATURE_NORM`, min/range → [0,1])

| # | Feature | min | range |
|---|---|---|---|
| 0 | prb_dl_pct | 0.0 | 100.0 |
| 1 | sinr_db | −5.0 | 35.0 |
| 2 | connected_ues | 0.0 | 800.0 |
| 3 | power_w | 0.0 | 1200.0 |
| 4 | packet_loss_pct | 0.0 | 5.0 |

### D.5 Inference behaviour

- Poll cadence `POLL_INTERVAL_SEC` (30 s in Docker); 6-step sliding window per cell.
- Rule-based fallback while the window fills (first ~60 s).
- AI action gate: acted on only if softmax confidence ≥ `MIN_CONFIDENCE` (0.70).
- Weights persisted to `MODEL_PATH` (`kpi_model.pt`); trained on first boot if absent.

### D.7 Rule-based fallback (while window < `SEQ_LEN`), evaluated in order

```
if   prb_dl_pct > OVERLOAD_PRB                                  → OVERLOAD
elif prb_dl_pct < UNDERLOAD_PRB                                 → UNDERLOAD
elif sinr_db    < SINR_MIN_DB                                   → SINR_LOW
elif power_w    > POWER_WASTE_W and connected_ues < POWER_WASTE_UE → POWER_WASTE
else                                                           → NORMAL
```

Act gate: an action fires iff `source == "RULE"` **or** `confidence ≥ MIN_CONFIDENCE`.
Cell KPIs are read from InfluxDB over `range(start: -3m)`, pivoted by field.
DU load map `du_avg` = mean `prb_dl_pct` per DU across current cells.

### D.8 SON actions per class

| Class | Action |
|---|---|
| OVERLOAD | Pick least-loaded DU where `du ≠ current` **and** `du_avg < OVERLOAD_PRB − 20`. Move the cell only if `now − last_moved[cell] ≥ POLL_SEC·3` (cooldown). Writes alert WARNING(OVERLOAD) + INFO(LOAD_BALANCE) and son_action `LOAD_BALANCE`. |
| UNDERLOAD | Alert INFO(UNDERLOAD); son_action `TRAFFIC_STEER` recommending handover to the min-PRB other DU (enable sleep/DTX). |
| SINR_LOW | Alert CRITICAL(SINR_DEGRADATION); best-effort `POST /son/pci-reopt` on Controller — reassigns the cell's PCI collision-free among its 3 km neighbours (see Appendix F.1); son_action `PCI_REOPT_REQUEST`. |
| POWER_WASTE | Alert WARNING(POWER_WASTE); son_action `DTX_RECOMMEND`, estimated saving = `power_w · 0.35` W. |

## Appendix E — Planning Engine Constants

### E.1 PCI planner (`pci_planner.py`)

`PCI_MAX = 1007`; `ADJACENCY_RADIUS_KM = 3.0` (Haversine). Greedy graph-colouring,
cells processed in **descending adjacency-degree** order. A PCI is valid for a cell if,
among assigned neighbours, it is **collision-free** (no shared PCI) **and confusion-free**
(no shared `PCI mod 3`); the smallest such PCI is chosen. **Fallback:** when ≥4 cells are
mutually adjacent, all three `mod-3` classes are occupied and no confusion-free PCI exists —
the planner then takes the smallest **collision-free** PCI (3GPP: collision forbidden,
confusion should-avoid). The Controller `POST /son/pci-reopt` SON action uses the same rule.

### E.2 Slice allocator (`slice_allocator.py`)

Minimum guaranteed PRB fraction `MIN_PRB`: eMBB 0.10, URLLC 0.05, mMTC 0.02.
Latency targets `LATENCY_TARGET_MS`: eMBB 30, URLLC 1, mMTC 100.
Algorithm: normalise traffic profile → enforce `MIN_PRB` floor → re-normalise to 1.0 →
`max_ues` per slice ∝ PRB fraction. Warns if `e2e_ms ≤ 1` and URLLC PRB < 0.15.

`timing_sync_strategy`: TDD bands = {n78, n41, n257, n258, n260, n261}.
`fronthaul_us ≤ 50` or any TDD → IEEE-1588-PTP-Class-C; `≤ 200` → Class-B; else SyncE.

### E.3 Heuristic placement & grouping (`placement.py`)

Costs: `COST_PER_CELL_USD = 50,000`, `COST_PER_DU_USD = 30,000`, `COST_PER_CU_USD = 80,000`.
Ranges: `FRONTHAUL_RADIUS_KM = 5.0`, `MIDHAUL_RADIUS_KM = 25.0`.
Defaults: `max_cells_per_du = 3`, `max_dus_per_cu = 4`.
`CANDIDATE_CELLS` (`placement.py`) — the 10 candidate macro sites used by the heuristic and
exposed via `GET /candidates`. All are seeded as `band="n78"`, `freq_mhz=3500`,
`generation="5G"`, `max_ues=900`; vendor round-robins via `_VENDOR_CYCLE`
(`[Nokia, Ericsson, Samsung, ZTE, Nokia, Ericsson, Samsung, ZTE, Nokia, Ericsson]`) and 5G
hardware fields come from `_HW_5G` (Nokia 1000/250/3800, Ericsson 950/**240**/3600,
Samsung 900/225/3400, ZTE 1000/250/3200 = tx_w/idle_w/peak_dl). Coordinates are within
±0.002° of the [B.1](#b1-site-coordinates-10-macro-towers) live sites:

```

```
Selection: rank `CANDIDATE_CELLS` by `score` descending, take the top `max_cells`.
DUs named `DU-BLR-NN`, CUs `CU-BLR-NN`; both grouped greedily by Haversine from the
highest-density anchor outward (a cell joins the nearest unfinished DU within
`FRONTHAUL_RADIUS_KM`, capped at `max_cells_per_du`). Earth radius `R = 6371 km`.
Latency estimates: fronthaul `dist_km·5 + 10` µs; midhaul `dist_km·0.01 + 0.5` ms.

> resolved decision, **simulators follow the applied plan's ids** ([Appendix C](#appendix-c--simulation-models)):
> deploy one DU container per plan-emitted `DU_ID`. `plan_to_topology()` does not remap names.

### E.4 MIP placement (`mip_placer.py`, `select_cells_mip`)

`install_cost_per_site = budget·0.6 / max_sites`; `op_cost = 1000` USD/period;
default `sinr_min_db = 10.0`, `time_limit_sec = 120`. Solver CBC via `pulp`;
on non-`Optimal` status, falls back to `select_cells(500, budget, bands)`
(`source = "heuristic_fallback"`).

**COST-231 Walfisch-Ikegami params** (`PropagationParams`): `h_tx = 25 m`, `h_rx = 2 m`,
`h_bld = 10 m`, `b_sep = 50 m`, `w_street = 25 m`, `phi = 30°`, `metropolitan = True`;
link budget `min_rx_power = −100 dBm`, `sinr_min = 10 dB`, `noise = −120 dBm`. Per-band
TX: n78/n41/n28 = 43 dBm / 18 dBi; B3/B40 = 40 dBm / 15 dBi.

**Decision variables** (all binary): `z[j,t]` build site j in period t, `y[j,t]` site j active
in period t, `x[i,j,t]` demand cluster i assigned to site j in period t (created only for
`(i,j)` pairs where j is in i's feasible set). **Feasible set** `S(i)` = candidate sites whose
received power at cluster i ≥ `min_rx_power_dbm` (−100), from `compute_link_powers()` using the
Walfisch-Ikegami params below. `S[j].max_channels` = the site's UE capacity; `dc.n_channels` ≈
the cluster's Erlangs.

**Constraint (8) linearisation — exact terms.** With `SINR_lin = 10^(sinr_min_db/10)` and
`P_noise = 10^(noise_power_dbm/10)` (both converted to mW; per-link powers `p_ij` also in mW):

```
α(i,t) = Σ_{j∈S(i)} p_ij · x[i,j,t]          # received power from the assigned BS
β(i,t) = Σ_{j∈S(i)} p_ij · y[j,t]            # total power from all active BSs reachable by i
constraint:  α(i,t)·(1 + SINR_lin) ≥ SINR_lin·P_noise + SINR_lin·β(i,t)
```

i.e. the assigned signal must beat noise plus interference (every other active reachable BS) at
the required SINR. One such constraint per (cluster i, period t) that has ≥1 feasible site.

**Demand clusters** (`BANGALORE_DEMAND_CLUSTERS`, one per DU zone — `n_channels` ≈ Erlangs):
`DC-MLS-N` (13.0070, 77.5700, 160), `DC-MLS-C` (13.0015, 77.5650, 150),
`DC-MLS-S` (12.9955, 77.5595, 150).

**Multi-period profiles**: Case A `[[DC-MLS-C], [DC-MLS-N, DC-MLS-S]]`;
Case B `[[DC-MLS-N], [DC-MLS-C, DC-MLS-S], [DC-MLS-N, DC-MLS-C, DC-MLS-S]]`.
